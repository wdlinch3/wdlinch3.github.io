Website Workflow Guide

Jekyll, Quarto, VS Code, Brave, Jupyter, and local preview servers

Author

William D. Linch III

Core principle

Do not preview everything the same way. Different source formats require different build and preview tools.

The practical rule is:

Task Edit with Preview with URL
Homepage VS Code Jekyll http://127.0.0.1:4000/
Normal website page VS Code Jekyll http://127.0.0.1:4000/path/page.html
Research page VS Code Jekyll http://127.0.0.1:4000/research/...
Quarto essay VS Code Quarto Quarto-provided localhost URL
Rendered Quarto page on site VS Code Jekyll http://127.0.0.1:4000/path/page.html
Notebook JupyterLab JupyterLab Jupyter URL
Plain static test HTML VS Code Python server or browser http://localhost:8000/ or file://
Public site check Browser GitHub Pages https://wdlinch3.github.io/

Short version:

Jekyll = preview the website.
Quarto = write/render documents.
VS Code = edit files.
Brave = view local server URLs.
Jupyter = notebooks only.
Python http.server = simple static files only.

Raw HTML files

Use this only for standalone HTML that does not depend on Jekyll.

Example:

A raw HTML file is safe to open directly in Brave only if it does not require:

Liquid syntax
Jekyll layouts
Jekyll includes
relative_url filters
site-wide injected CSS
site-wide injected MathJax

Opening directly in Brave gives a URL like:

file:///Users/wdlinch3/Documents/github/wdlinch3.github.io/index.html

Quarto writing pages

Use this for .qmd files such as:

talks_and_lectures/writings/once_upon_a_time.qmd
vlo_aa_ss.qmd
vlo_aa_u.qmd
vlo_aa_x.qmd

Quarto files are source files. A browser does not understand .qmd directly. They must be rendered by Quarto.

Preview a single Quarto file:

cd ~/Documents/github/wdlinch3.github.io
quarto preview talks_and_lectures/writings/once_upon_a_time.qmd

Quarto will open a local server. The port may be something like:

http://localhost:6482

The exact port can vary.

Render the file to HTML:

quarto render talks_and_lectures/writings/once_upon_a_time.qmd

This produces:

talks_and_lectures/writings/once_upon_a_time.html

Then preview the rendered HTML as part of the Jekyll site:

bundle exec jekyll serve --livereload

Open:

http://127.0.0.1:4000/talks_and_lectures/writings/once_upon_a_time.html

Use this workflow for:

math-heavy essays
lecture notes
standards/VLO documents
converted documents
documents that should eventually render to both HTML and PDF

Typical .qmd front matter:

---
title: "Once Upon a Time"
format:
  html:
    theme: darkly
    html-math-method: mathjax
    toc: false
    css:
      - ../../assets/css/style.css
---

If adding a Quarto-specific stylesheet later:

---
title: "Once Upon a Time"
format:
  html:
    theme: darkly
    html-math-method: mathjax
    toc: false
    css:
      - ../../assets/css/style.css
      - ../../assets/css/quarto_site.css
---

Important distinction:

quarto preview file.qmd     previews the Quarto document
jekyll serve                previews the full website

Markdown and Quarto Markdown

Plain Markdown files use extension:

.md

Quarto Markdown files use extension:

.qmd

For the current website/document workflow, prefer:

.qmd

when the document is intended to become a polished web or PDF page with math, tables, formatting, and possible future publication.

Use .md for simpler project notes, README files, GitHub-facing documentation, or files that do not need Quarto features.

Markdown preview in VS Code is fine for rough editing, but it is not the final website preview.

Correct previews:

README.md                  → VS Code Markdown preview is okay
ordinary Jekyll .md page   → Jekyll preview
Quarto .qmd page           → Quarto preview

Liquid syntax

Liquid is the templating language Jekyll uses.

Example:

{{ '/assets/css/style.css' | relative_url }}

This means:

Take /assets/css/style.css
adjust it for the site base URL
insert the result into the HTML

The double braces mean “evaluate this expression and insert the result here”:

{{ ... }}

The vertical bar applies a filter:

| relative_url

Use Liquid inside Jekyll-processed files:

_layouts/default.html
_includes/*.html
index.html
research/*.html
aa/*.html
pages with YAML front matter

Use it for internal site links and assets:

<link rel="stylesheet" href="{{ '/assets/css/style.css' | relative_url }}">

<a href="{{ '/research/effective_actions.html' | relative_url }}">Effective actions</a>

<img src="{{ '/assets/images/backgrounds/EagleIR.jpeg' | relative_url }}" alt="">

Use relative_url for ordinary internal links and assets.

Use absolute_url only when a full public URL is genuinely needed:

{{ '/research/effective_actions.html' | absolute_url }}

Good uses of absolute_url include:

canonical URLs
RSS feeds
Open Graph metadata
sitemaps
external sharing metadata

Do not expect Liquid to work inside files that Jekyll does not process, such as ordinary CSS or JavaScript files.

Internal and external links

Internal links to your own site should generally use Liquid inside Jekyll files:

<a href="{{ '/cv/cv.html' | relative_url }}">CV</a>
<a href="{{ '/aa/' | relative_url }}">Advanced Astronomy</a>
<a href="{{ '/talks_and_lectures/' | relative_url }}">Talks and lectures</a>

External links should be hard-coded:

<a href="https://en.wikipedia.org/wiki/Thomas_Jefferson_High_School_for_Science_and_Technology">TJHSST</a>
<a href="https://arxiv.org/">arXiv</a>

PDFs or files hosted in your own repository should use relative_url in Jekyll files:

<a href="{{ '/cv/wdlinch_cv.pdf' | relative_url }}">PDF CV</a>

PDFs or files hosted externally should use a full hard-coded URL:

<a href="https://arxiv.org/pdf/hep-th/0608005">Paper PDF</a>

Rule:

Own site, own files, own assets    → use {{ '...' | relative_url }} in Jekyll files
Outside website                    → hard-code the full https://... URL
Canonical/share/feed metadata      → use {{ '...' | absolute_url }}

CSS files

The main CSS files live in:

assets/css/

Currently relevant files include:

assets/css/style.css
assets/css/research.css
assets/css/aa.css

For Jekyll layouts, include CSS using:

<link rel="stylesheet" href="{{ '/assets/css/style.css' | relative_url }}">

For Quarto .qmd files, include CSS in the YAML front matter using a relative filesystem path from the .qmd file:

css:
  - ../../assets/css/style.css

Do not use Liquid inside Quarto YAML unless deliberately running the output through Jekyll as a second processing stage. For now, avoid that.

To test whether CSS is being served by Jekyll:

curl -I http://127.0.0.1:4000/assets/css/style.css

Expected result:

HTTP/1.1 200 OK
Content-Type: text/css

To see what CSS a page is linking:

curl -s http://127.0.0.1:4000/ | grep -i stylesheet

MathJax

MathJax renders LaTeX-style math in the browser.

For Jekyll pages, MathJax should be included globally through the site layout or an include.

For Quarto pages, use:

format:
  html:
    html-math-method: mathjax

Use ordinary inline math:

$E = mc^2$

Use display math:

$$
S = \int d^4x \sqrt{-g}\,\mathcal{L}
$$

If math works in Quarto preview but not in the Jekyll-served page, check whether the rendered HTML includes MathJax or whether Quarto used a different math-rendering method.

Browser preview

Use Brave as the browser previewer, but open local server URLs, not source files.

Correct:

http://127.0.0.1:4000/
http://127.0.0.1:4000/research/effective_actions.html
http://127.0.0.1:4000/talks_and_lectures/writings/once_upon_a_time.html

Usually wrong:

Why it is wrong:

Brave does not run Jekyll.
Brave does not process Liquid.
Brave does not apply layouts or includes.
Brave only renders the raw file.

Port numbers

Several local servers may appear.

Port 4000

Usually Jekyll:

bundle exec jekyll serve --livereload

Open:

http://127.0.0.1:4000/

Quarto may also sometimes use 4000 if it is available, so check the terminal to see what server is actually running.

To see what is using port 4000:

lsof -i :4000

If it says ruby, it is probably Jekyll.

To stop it from the terminal where it is running:

Ctrl + C

To kill it from another terminal:

kill $(lsof -ti :4000)

Port 8000

Usually Python’s simple static server:

python3 -m http.server 8000

Open:

http://localhost:8000/

This is useful only for plain static files. It does not run Jekyll or Quarto.

For the main website, this is generally not the right tool.

To see what is using port 8000:

lsof -i :8000

To kill it:

kill $(lsof -ti :8000)

Port 3000

Often used by VS Code Live Preview or other development servers.

For this repository, do not rely on it unless deliberately using Live Preview.

Random Quarto ports

Quarto may use ports like:

http://localhost:6482

That is normal. The terminal output from quarto preview tells you the correct URL.

Common problems and fixes

Page loads but CSS is ignored

Check whether the CSS file is reachable:

curl -I http://127.0.0.1:4000/assets/css/style.css

Check whether the page links the CSS:

curl -s http://127.0.0.1:4000/ | grep -i stylesheet

If CSS exists but the page looks stale, hard refresh Brave:

Cmd + Shift + R

Site looks wrong when opening index.html directly

Reason:

You are opening the Jekyll source file directly.

Fix:

bundle exec jekyll serve --livereload

Then open:

http://127.0.0.1:4000/

Address already in use

Example:

OSError: [Errno 48] Address already in use

Find the process:

lsof -i :4000

or:

lsof -i :8000

Kill it:

kill $(lsof -ti :4000)

or:

kill $(lsof -ti :8000)

Quarto preview works, but the page does not look like the website

Reason:

Quarto preview is showing a standalone Quarto document.
It is not using the full Jekyll layout.

Fix options:

Accept Quarto pages as standalone essays.
Add site CSS to the Quarto YAML.
Add a Quarto-specific stylesheet.
Later, integrate Quarto output more deeply into Jekyll if desired.

Hard-coded links broke local preview

Avoid hard-coding internal site paths like:

<link rel="stylesheet" href="https://wdlinch3.github.io/assets/css/style.css">

or:

<link rel="stylesheet" href="/wdlinch3.github.io/assets/css/style.css">

Use Liquid in Jekyll files:

<link rel="stylesheet" href="{{ '/assets/css/style.css' | relative_url }}">

Use normal relative paths in Quarto files:

css:
  - ../../assets/css/style.css

Files and folders not to edit

Do not edit:

_site/

That is generated output from Jekyll.

Do not commit _site/ unless there is a very specific reason. For a normal GitHub Pages/Jekyll repository, source files should be committed, not generated _site output.

Final decision rule

Editing a normal website page?       Run Jekyll.
Editing a .qmd writing/math page?    Run Quarto.
Checking the whole site shell?       Run Jekyll.
Publishing?                          Commit source + rendered HTML as needed.
